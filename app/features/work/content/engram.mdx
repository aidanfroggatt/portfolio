  <WorkProse>
    I built Engram as a direct reaction to the friction of modern social media. I wanted to preserve
    the habit of keeping a digital journal, but I was exhausted by performative algorithms, forced
    engagement loops, and inevitable data lock-in.

    For years, the path of least resistance for digital scrapbooking was maintaining a private
    Instagram account. While frictionless, it inherently meant sacrificing data ownership. Platforms

    engineered Engram to reclaim that space. I designed it to deliver the fluid, instant temporal
    navigation of a native mobile application, but anchored by a zero-trust, heavily cached
    distributed architecture.

<WorkSection>
  <WorkHeader type="INFRASTRUCTURE" title="Decoupling Read and Write Paths." />
  <WorkSplitContent subtitle="Designing for asymmetrical data flows">
    When building a media vault, the foundational architecture is driven by a simple fact of physics: media files are heavy. I needed the system to handle massive, batch-uploaded batches directly from mobile devices without choking serverless functions or blowing up memory limits. 
    
    To solve this, I violently separated the Read Path from the Write Path. By treating media delivery and metadata orchestration as two entirely isolated systems, the architecture can scale independently based on the specific load requirements of the user.
  </WorkSplitContent>

<WorkSection>
  <WorkHeader type="SECURITY" title="The Identity Valve." />

<WorkProse>
  Migrating media delivery to the edge solved the bandwidth problem, but it introduced immense
  security friction. A multi-tenant application cannot blindly trust client-side claims, and leaving
  a deep storage bucket exposed to the public internet was a non-starter. While Clerk manages
  session lifecycles on the frontend, I engineered "The Identity Valve" inside the Cloudflare Worker
  to enforce an absolute zero-trust perimeter. The edge script dynamically fetches and caches
  Clerk's public JSON Web Key Set (JWKS), mathematically verifying the cryptographic signature of
  the user's incoming JWT before a single byte of media is ever requested.
</WorkProse>

<WorkCallout title="Path Partitioning">
  Authentication is meaningless without authorization. Building the proxy required navigating the
  notoriously fragile AWS V4 Signature protocol. Once authenticated, the Edge Worker enforces strict
  Path Partitioning. It guarantees that even if a user possesses a valid token, they can only ever
  resolve the S3 object prefix matching their unique User ID, completely neutralizing horizontal
  object enumeration.
</WorkCallout>

<WorkSection>
  <WorkHeader type="DATA LAYER" title="Stateful Ingestion." />

<WorkSplitContent subtitle="The Three-Phase Commit Handshake">
  While the Read Path is stateless and heavily cached, the Write Path demands absolute relational
  consistency. A "fire and forget" upload model creates orphan files—assets sitting in cloud
  storage without a matching metadata record in the database. To solve this, I built the ingestion
  pipeline around a strict, three-phase transactional handshake.

I integrated the Ent ORM to manage the Neon Postgres instance, treating the database as a living
schema defined strictly in Go. By leveraging Ent's `SetNillable` patterns, the application
cleanly handles the optionality of spatial data (latitude and longitude) extracted from EXIF
metadata without breaking strict type boundaries.

</WorkSplitContent>

{/* PLACEHOLDER: UI Animation showing the 3 phases of upload: Presigning lock, Streaming progress bar, and Database commit checkmark */}

  <WorkFeatureList>
    <WorkFeature title="1. Intent Authorization">
      The Next.js client requests an upload intent. The Go backend authorizes the user context and generates a restricted, short-lived B2 PUT URL.
    </WorkFeature>
    <WorkFeature title="2. Direct-to-Cloud Streaming">
      The frontend streams the raw binary payload directly to Backblaze B2. Bypassing the Go API prevents memory exhaustion and timeout errors during massive batch uploads.
    </WorkFeature>
    <WorkFeature title="3. Transactional Commit">
      Only after Backblaze confirms a successful HTTP 200 upload event does the client trigger the commit endpoint. Go pulls the verified `user_id` from the secure context, merges it with the client-extracted EXIF data, and persists the final record via Ent.
    </WorkFeature>
  </WorkFeatureList>
</WorkSection>

  <WorkProse>
    To support this fluid chronological experience, I implemented a memoized grouping engine using `date-fns`. Assets are mathematically grouped by ISO-8601 timestamps, allowing users to instantly swap between "Day" and "Year" views without triggering additional network round-trips. 
    
    I also engineered a conditional hydration layer to solve the "Broken Video" problem. The renderer dynamically swaps between the optimized Next.js `<Image />` component and raw HTML5 `<video>` tags based on the MIME type, utilizing the `#t=0.001` URL suffix trick to force the browser to generate a lightweight poster thumbnail from the first frame.

<WorkSection>
  <WorkHeader type="DATA PIPELINE" title="Real-World Data is Messy." />

<WorkSplitContent subtitle="Recursive filesystem ingestion">
  The decision to migrate a decade of my own memories off of Meta's servers provided the ultimate
  reality check for the ingestion engine. The initial plan—relying on a clean JSON-based metadata
  import—was shattered by the chaotic reality of the actual data dump, which was a labyrinth of
  nested folders, missing manifests, and stripped EXIF data.

Rather than writing brittle scripts to fix Meta's export, I pivoted the architecture to rely on
a Recursive Filesystem Walker. I engineered the pipeline to fall back on deriving temporal
context directly from the folder hierarchy itself (e.g., `/202405/`) when EXIF data failed,
proving the system's resiliency.

</WorkSplitContent>

<WorkSection>
  <WorkHeader type="DEVOPS" title="Taming the Polyglot Monorepo." />

<WorkProse>
  When you start a polyglot monorepo, the promise is a unified workflow. The reality is often "Death
  by a Thousand Configs." Architecting a codebase that gracefully handles Next.js frontends,
  Cloudflare Workers, and Go binaries required establishing a mathematically reproducible build
  pipeline. Turborepo acts as my master orchestrator, but strict environment isolation presented
  unique hurdles—specifically, Turborepo stripping the OS-level variables required by the Go
  compiler on Windows to locate its build cache. Explicitly whitelisting these paths via
  `turbo.json` allowed me to maintain lightning-fast, parallelized builds locally without
  compromising my hermetic CI environments in the cloud.
</WorkProse>

<WorkInsight title="Infrastructure as a Product">
  The most profound paradigm shift during the development of Engram was treating infrastructure not
  as a deployment afterthought, but as the core product feature. Building a button that uploads a
  photo is trivial. Building a system that can sustain 10,000 parallel uploads securely, parse the
  resulting metadata accurately, and serve that media globally with zero egress cost requires
  rigorous system design. By enforcing SSH commit signing, migrating DNS authority to Cloudflare for
  a unified namespace, and co-locating edge logic tests via Vitest to mathematically prove my
  authorization partitioning, Engram matured from a conceptual CRUD application into a hardened
  archival engine. It is ready for the next 10,000 memories.
</WorkInsight>
